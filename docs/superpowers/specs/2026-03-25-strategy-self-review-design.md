# Strategy Self-Review System — Design Spec

**Date:** 2026-03-25
**Scope:** Alert outcome tracking + weekly AI strategy review
**Approach:** Daily cron updates alert outcomes, weekly cron generates AI review report

---

## 1. Problem Statement

The current system generates trading lessons only when tracked trades fail or expire. It has no mechanism to:
- Verify whether alerts that triggered actually led to profitable moves
- Identify stocks that rose significantly but were not selected by AI (missed opportunities)
- Understand systemic patterns in AI's decision-making blind spots
- Generate periodic strategy reviews for the user to discuss and act on

---

## 2. Database Layer

### 2.1 New Migration: `010_strategy_review.sql`

#### `alert_outcomes` table

Tracks the price performance of every alert after 1, 3, and 7 days.

```sql
CREATE TABLE IF NOT EXISTS alert_outcomes (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  alert_id uuid NOT NULL REFERENCES alerts(id) ON DELETE CASCADE UNIQUE,
  user_id uuid NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  ticker text NOT NULL,
  alert_price numeric(12,4) NOT NULL,
  alert_date timestamptz NOT NULL,
  ai_recommendation text,        -- BUY, SELL, HOLD, or NULL if no analysis
  ai_confidence integer,          -- 0-100 or NULL
  ai_reasoning text,              -- denormalized from analyses.summary
  price_after_1d numeric(12,4),
  price_after_3d numeric(12,4),
  price_after_7d numeric(12,4),
  change_pct_1d numeric(8,2),
  change_pct_3d numeric(8,2),
  change_pct_7d numeric(8,2),
  outcome_category text CHECK (outcome_category IN ('hit', 'miss', 'marginal', 'missed_opportunity', 'correct_skip')),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

CREATE TRIGGER alert_outcomes_updated_at
  BEFORE UPDATE ON alert_outcomes
  FOR EACH ROW EXECUTE FUNCTION update_updated_at();

CREATE INDEX IF NOT EXISTS idx_alert_outcomes_user
  ON alert_outcomes (user_id, alert_date DESC);

CREATE INDEX IF NOT EXISTS idx_alert_outcomes_category
  ON alert_outcomes (outcome_category)
  WHERE outcome_category IS NOT NULL;

ALTER TABLE alert_outcomes ENABLE ROW LEVEL SECURITY;

CREATE POLICY alert_outcomes_select ON alert_outcomes
  FOR SELECT USING (user_id = auth.uid());
```

**Outcome classification rules** (applied when `price_after_7d` is filled):

BUY-centric analysis only. SELL recommendations are excluded from outcome tracking (the system is designed for identifying buying opportunities, not shorting).

- **hit**: `ai_recommendation = 'BUY'` AND `change_pct_7d >= 2`
- **marginal**: `ai_recommendation = 'BUY'` AND `0 <= change_pct_7d < 2` (directionally correct but small gain)
- **miss**: `ai_recommendation = 'BUY'` AND `change_pct_7d < 0`
- **missed_opportunity**: `ai_recommendation IN ('HOLD', NULL)` AND `change_pct_7d >= 5`
- **correct_skip**: everything else (HOLD/NULL and stock didn't rise significantly, or SELL recommendations)

Note: SELL recommendations are always classified as `correct_skip` regardless of outcome, since the system focuses on buy-side accuracy.

#### `strategy_reviews` table

Weekly review reports generated by AI.

```sql
CREATE TABLE IF NOT EXISTS strategy_reviews (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id uuid NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  review_period_start date NOT NULL,
  review_period_end date NOT NULL,
  total_alerts integer NOT NULL DEFAULT 0,
  hits integer NOT NULL DEFAULT 0,
  misses integer NOT NULL DEFAULT 0,
  missed_opportunities integer NOT NULL DEFAULT 0,
  correct_skips integer NOT NULL DEFAULT 0,
  hit_rate numeric(5,2) DEFAULT 0,
  missed_opportunity_rate numeric(5,2) DEFAULT 0,
  top_missed jsonb,             -- [{ticker, change_pct_7d, reason}]
  top_misses jsonb,             -- [{ticker, change_pct_7d, ai_confidence}]
  ai_analysis text,             -- AI-generated review (Chinese)
  recommendations jsonb,         -- [{suggestion, reasoning, priority}]
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX IF NOT EXISTS idx_strategy_reviews_unique_period
  ON strategy_reviews (user_id, review_period_start, review_period_end);

CREATE INDEX IF NOT EXISTS idx_strategy_reviews_user
  ON strategy_reviews (user_id, review_period_end DESC);

ALTER TABLE strategy_reviews ENABLE ROW LEVEL SECURITY;

CREATE POLICY strategy_reviews_select ON strategy_reviews
  FOR SELECT USING (user_id = auth.uid());
```

## 3. Backend Layer

### 3.1 New Module: `api/_lib/outcome-tracker.ts`

Two exported functions:

#### `updateAlertOutcomes()`

Called daily from existing `check-trades.ts` cron.

```
1. Find alerts that are 1, 3, or 7 days old and don't have the corresponding price filled:
   - 1 day old: price_after_1d IS NULL
   - 3 days old: price_after_3d IS NULL
   - 7 days old: price_after_7d IS NULL

2. For alerts without an alert_outcomes row yet (new alerts from yesterday):
   - Create alert_outcomes row with alert_price, alert_date, ai_recommendation, ai_confidence
   - Source ai_recommendation/confidence from the analyses table joined via alert_id

3. Collect unique tickers from all pending outcomes (deduplicate — same ticker may appear at 1d, 3d, and 7d)
4. Fetch prices in batches of 5 (to avoid Yahoo Finance rate limiting), using existing `cached()` wrapper for dedup
5. Update the appropriate price column and calculate change_pct

6. For rows where price_after_7d is now filled:
   - Apply classification rules → set outcome_category
```

**Day calculation:** Uses calendar days (not business days). An alert from Monday gets its 1d check on Tuesday, 3d on Thursday, 7d on next Monday. Weekend/holiday prices return the last trading day's close, which is acceptable for this analysis. Simplicity over precision — the goal is trend direction, not exact timing.

#### `generateWeeklyReview(userId: string)`

Called from weekly cron.

```
1. Query alert_outcomes where updated_at is within the review period (last 7 days) AND outcome_category IS NOT NULL
   (This captures outcomes that were just classified, meaning alerts from ~14 days ago that now have 7d data)
2. Count: hits, misses, missed_opportunities, correct_skips
3. For missed_opportunities: query original alert + analysis to understand why AI passed
   - If no analysis exists: "此 alert 無分析記錄"
   - If HOLD: include AI's reasoning (summary field)
   - If confidence was below threshold: note the confidence score
4. For misses: gather the AI's original confidence and reasoning
5. Build a context string with all this data
6. Call Claude AI with a review prompt:

   Prompt structure:
   - System: "你是一個交易策略檢討分析師"
   - Context: all outcome data, missed opportunities details, miss details
   - Task: "分析以下一週交易數據，找出 AI 的系統性盲點和改進方向"
   - Output format: JSON with ai_analysis (text), recommendations (array of {suggestion, reasoning, priority})

7. Insert into strategy_reviews table
8. Send Telegram notification with summary
```

### 3.2 Cron Integration

**Existing `api/cron/check-trades.ts` — add at end:**

```typescript
// Update alert outcomes (1d/3d/7d price tracking)
try {
  const outcomeResults = await updateAlertOutcomes()
  console.log(`[OUTCOMES] Updated: ${outcomeResults.updated}, Classified: ${outcomeResults.classified}`)
} catch (err: any) {
  console.error('[OUTCOMES] Error:', err.message)
}
```

**New `api/cron/weekly-review.ts`:**

Triggered via Vercel cron on Sundays at 10 AM EST.

```typescript
// vercel.json cron config:
// { "path": "/api/cron/weekly-review", "schedule": "0 15 * * 0" }
// (15:00 UTC = 10:00 EST)
```

Endpoint:
1. Validate cron secret
2. Get all users with active portfolios
3. For each user: `await generateWeeklyReview(userId)`
4. Return summary

### 3.3 Webhook Integration

In `api/webhook.ts`, after saving the alert and analysis, create the initial `alert_outcomes` row:

```typescript
// After alertRecord is saved and analysis is done:
await adminClient.from('alert_outcomes').upsert({
  alert_id: alertRecord.id,
  user_id: userId,
  ticker,
  alert_price: parseFloat(price) || 0,
  alert_date: new Date().toISOString(),
  ai_recommendation: analysis.recommendation,
  ai_confidence: analysis.confidence,
  ai_reasoning: analysis.summary || null,
}, { onConflict: 'alert_id', ignoreDuplicates: true })
```

This ensures every alert gets an outcome row immediately, and the daily cron just fills in price data.

## 4. Frontend Layer

### 4.1 New Page: `/reviews` (ReviewsPage.tsx)

Or extend the existing StrategyPage (`/strategy`). Use a new route `/reviews` to keep things clean.

**Page structure:**

#### Header
- Title: "策略檢討" (serif-heading, large)
- Subtitle: "AI 每週自動分析交易表現，找出盲點與改進方向"

#### Summary Stats (current week)
- 4 stat cards (matching DashboardPage pattern):
  - 總 alert 數
  - 命中率 (hits / total BUY recommendations)
  - 漏掉率 (missed_opportunities / total non-BUY)
  - 正確忽略率

#### Weekly Reports List
- Cards ordered by `review_period_end DESC`
- Each card shows:
  - Week range (e.g., "3/17 - 3/23")
  - Hit/miss/missed stats as pills
  - AI analysis preview (first 2 lines, expandable)
  - Click to expand: full AI analysis + recommendations list + missed opportunity details

#### Missed Opportunities Detail (inside expanded report)
- Table showing stocks that rose but AI didn't pick:
  - Ticker, alert price, price after 7d, change %, AI's original recommendation, reason AI passed

#### Recommendations Section (inside expanded report)
- List of AI suggestions with priority badges (高/中/低)
- Each recommendation: suggestion text + reasoning

## 5. AI Review Prompt

The weekly review prompt sent to Claude:

```
你是一個資深交易策略檢討分析師。以下是本週（{start} 至 {end}）的 AI 交易表現數據。

## 統計摘要
- 總 alert 數: {total}
- 命中 (BUY 且漲 ≥2%): {hits} ({hit_rate}%)
- 誤判 (BUY 且跌): {misses} ({miss_rate}%)
- 漏掉 (未選但漲 ≥5%): {missed_opportunities} ({missed_rate}%)
- 正確忽略: {correct_skips}

## 漏掉的機會（詳細）
{foreach missed_opportunity:}
- {ticker}: alert 價格 ${alert_price} → 7天後 ${price_after_7d} (漲 {change_pct_7d}%)
  AI 當時判斷: {ai_recommendation} (信心度 {ai_confidence}%)
  AI 原因: {ai_reasoning || '無分析記錄'}

## 誤判的交易（詳細）
{foreach miss:}
- {ticker}: alert 價格 ${alert_price} → 7天後 ${price_after_7d} (跌 {change_pct_7d}%)
  AI 信心度: {ai_confidence}%

## 請分析（繁體中文，JSON 格式）：
{
  "ai_analysis": "2-3段完整分析：AI 本週的表現概述、系統性盲點分析、最大的改進空間",
  "recommendations": [
    {
      "suggestion": "具體可操作的建議（例如：降低 RSI > 75 股票的信心度扣分力度）",
      "reasoning": "為什麼這個改變會有幫助",
      "priority": "high | medium | low"
    }
  ]
}
```

## 7. Verification Plan

### DB
- Run migration, verify tables and indexes exist
- Insert test alert_outcomes rows, verify RLS restricts cross-user access

### Backend
- Trigger daily cron → verify alert_outcomes rows get price_after_1d filled for yesterday's alerts
- Wait (or simulate) 7 days → verify outcome_category is calculated
- Trigger weekly cron → verify strategy_reviews row is created with AI analysis
- Verify Telegram notification is sent

### Frontend
- Navigate to `/reviews` → verify page renders with empty state
- After a weekly review exists → verify report card displays
- Expand report → verify AI analysis and recommendations render
- Verify missed opportunities table shows correct data

### Edge Cases
- Alert with no analysis (webhook failed mid-way) → outcome should still be created, ai_recommendation = NULL
- Zero alerts in a week → skip review generation (don't create empty report)
- All alerts are correct_skip → report should note "本週表現穩定，無明顯漏掉的機會"
- Market holiday → price fetch returns stale data, should still work
